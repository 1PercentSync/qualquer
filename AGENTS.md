# Qualquer 渲染器

基于 CUDA + OptiX 的 Path Tracer。

- 设计文档索引：`docs/agent-context.md`
- 任务规划：`tasks/`

## 文档规范

- 文档内容：中文（`.md` 项目/设计文档；代码内文档见「代码文档」，用英文）
- 文档文件名：英文
- 规则与约束性表述直接陈述，不举例，保持抽象稳定。

### Agent 约束

#### 工具与执行边界

- **禁止**使用 Subagent（Agent 工具）或 Plan 工具，除非用户主动要求
- **禁止**执行 cmake、build、run 命令
- **创建和编辑文件时，只能**操作 C++/CUDA 源码文件、文档及构建配置
- 需要验证时告知用户在 CLion 中操作
- **禁止**例行 Git 检查；仅在出现明确风险信号或用户要求时，才针对具体文件运行最小必要检查
- **文件搜索边界**：搜索根必须限定在明确的代码库根目录内，禁止以 `/`、`~` 或未确认为代码库的大范围目录为根。允许根：当前工作目录、工作流指定的参考项目根。定位 SDK/系统头文件时不扫文件系统，通过项目配置解析其实际路径

#### 信息呈现

- **工具结果非展示**：工具返回值仅供 agent 使用，不视为已向用户展示。不得靠执行工具并假定用户已看到其输出；需用户知晓的内容必须在 agent 文字中用自己的话说明
- **用户只读正文**：用户阅读的是 agent 的叙述正文；工具调用之间可以输出内容，但不得把仅出现在调用间隙、易被略过的碎片当作用户已读。任何需用户获知的信息必须在正文中完整给出

#### 规划与设计

- **禁止**在未经确认时推断项目规划；须先查 `docs/` 确认后再陈述
- **文档非圣旨**：实现文档方案连续受阻、或方案复杂度与收益失衡时，按 `architecture.md` 的「决策可更新性 / 复杂度收益」原则回头检验方案，与用户讨论是否更新文档；不得把文档当作不可修改的权威硬干
- **设计评估义务**：陈述设计选择时，每条理由区分「依据」（引用文档或代码位置）与「判断」（基于事实的推断并标注）。不得编造项目中不存在的约束或需求。发现任务清单设计可能有局限时，主动提出
- **"参考"非"照搬"**：落地方式不唯一时，先澄清参考维度与取舍，确认方向后再深入；与本项目技术栈或复杂度收益不符的部分，主动提出差异和替代方案

#### 受阻处理

- **遇阻即停**：工具不可用、路径/权限问题等连续受阻时立即停下报告；调研或实验前先确认方向与范围；不反复试错；临时产物须清理

### 工作流程

#### 首次实现前复述

每次会话中，用户第一次要求开始实现任务时，必须先**完整复述**本节工作流程，然后**进入实现环节**；不得将复述理解为需要等待用户额外确认。

#### 一小项主循环

**一小项**：`tasks/` 清单中每一个 `- [ ]` 复选框条目。每完成一个须暂停等待用户审查，严禁合并多个复选框为一次完成。

1. 实现一小项 → **暂停**，等待用户审查
2. 按「授权」执行后续动作（见下）：
   - 用户回复**「继续」**：验收通过 → 代码与文档分开 commit → 更新 `tasks/` 与 `docs/agent-context.md`（独立 docs commit）→ 每次 commit 后立即 push → **开始下一项**
   - 用户仅表达**提交意图**：验收通过 → 代码与文档分开 commit → 更新进度文档并 push → **停下**，不开始下一项，等待下一条指令
3. 编译：不要求每一小项都能编译，只需一个 Step 结束时编译通过即可

代码 commit 与文档 commit 必须分离；每次 commit 后立即 push，不得积压，不得单独询问是否 push。

#### 授权

授权 commit 即同时授权 push（原子操作）。严禁自行推断指令隐含提交授权。

| 用户意图 | 可执行 | 不可执行 |
|----------|--------|----------|
| **继续** | 对已完成待审小项：验收通过 → 提交 → 更新 `tasks/` 与 `agent-context` → push → **开始下一项**。不授权用户尚未显式认可的额外改动 | 将「继续」扩大到未认可的额外改动 |
| **显式提交意图** | 对当前已完成工作：验收通过 → commit + push（含必要的 docs 进度更新） | **开始下一项**；把「提交」当成完整主循环推进 |
| 其他指令 | 只做该动作本身；完成后报告「已完成，未提交」并停下 | commit / push / 开始下一项 |

**疑问 / 质疑 / 探询不构成执行许可**，必须先据实回答 / 讨论。若疑问指向的内容本就在后续复选框，据实说明归属与顺序，**不得**借此提前或并入当前项，也**不得**据此提交或推进。

#### Step 实现前讨论

每个 **Step** 进入实现前须完成本节；讨论为交互节点（**须用户认可计划**），不得讨论完直接实现。粒度是 **Step**，不是 checkbox：同一 Step 内首项讨论经认可后，后续 checkbox 直接执行；进入下一 Step 再重新走本节。

**外部依据**（按任务需要取用，非每项必有）：参考项目代码、依赖库与 SDK 文档/头文件与示例、论文与技术报告。路径与范围以路线图、当前阶段文档或用户指定为准；不得无边界地扫盘。

1. **先读本仓库**：从任务清单提取本 Step 涉及的模块，建立「已有什么、缺什么」
2. **再读外部依据**（若有）：在本仓库认知之上对照实现、API 约束或算法表述；在回复中陈述读了哪些来源/区域、判定及依据
3. **与用户讨论计划**（须认可后再实现）：
   - **外部依据中有可对照的实现或规范**：三点——（1）依据侧做了什么 / 规定什么、为何；（2）本仓库怎么做；（3）精简、适应或偏离及理由
   - **无外部依据，或无可对照内容**：陈述本仓库实现计划与取舍依据，讨论聚焦自身方案

### 上下文管理

- **会话开始**：阅读 `docs/agent-context.md`，根据其中的必读列表加载所需文档

### Commit 规范

格式：`<type>(<scope>): <description>`

| type       | 用途            |
|------------|---------------|
| `feat`     | 新功能 / 新代码     |
| `docs`     | 文档            |
| `chore`    | 构建配置、工具链、杂项维护 |
| `fix`      | 修复 bug        |
| `refactor` | 重构（不改变行为）     |

scope 可选，使用架构层名

**语言**：description 用英文；type/scope 已是英文，保持一致

**分离原则**：代码变更和文档变更必须分开提交，不得混在同一个 commit 中

### 文档归档规范

Phase 切换或文档被替换时，旧文档必须先归档再创建新文档：

1. 使用 `git mv` 将旧文档移入 `docs/archive/` 或 `tasks/archive/`（保留 git 跟踪历史）
2. 归档作为**独立 commit**，不与新文档创建混在一起
3. `docs/` 和 `tasks/` 下的文件**成对归档**
4. 归档后更新 `docs/agent-context.md` 的归档文档列表

## 编码规范

### 头文件

使用 `#pragma once`，路径带项目前缀：`#include <qualquer/<layer>/xxx.h>`

### Device 头文件（.cuh）

CLion Nova 引擎只对被 `.cu` 文件 include 的 `.cuh` 赋予 CUDA 编译上下文。新建 `.cuh` 后必须在 `programs.cu` 中添加对应
`#include`，否则 Nova 无法解析 CUDA 内建符号，产生 `Cannot resolve symbol` 警告。

### 代码文档

- 格式：Javadoc 风格 Doxygen（`/** */`）
- 语言：英文
- `.h` + `.cpp` 配对：`.h` 写接口文档（**what**：做什么、语义、约束），`.cpp` 写实现注释（**why/how**：为什么这么做、算法解释）
- 仅 `.h`（纯头文件库 / 数据结构定义）：文档全部写在 `.h` 中，同时包含 what 和 why
- 仅 `.cpp`：文档全部写在 `.cpp` 中，同时包含 what 和 why
- 所有公开和私有接口、字段、枚举值均需写文档
- 文件头文档（`@file`）只写不会随开发变化的内容（文件名、所属模块），不写功能描述
- 全文禁用过时措辞：开发阶段名、未来时态
- 去冗余：不写解释代码逻辑的注释；注释只保留 why——为什么这么做、非显然语义与设计约束，代码本身已表达的无需再说
- 代码文件中不引用项目文档（`docs/`、`tasks/` 路径）；代码注释必须自包含，读者不需要查阅项目文档即可理解接口

### 控制流

- `if`、`else`、`for`、`while` 等语句体即使只有一行也**必须**使用 `{}`，不得省略

### 函数参数

- **不用默认参数**：函数声明不设默认值，调用方显式传值。避免调用点隐式依赖默认值、签名变更时遗漏实参。

### 日志

- 日志级别按语义选择，不为迁就当前 `kLogLevel` 而提升级别
- `kLogLevel` 控制运行时过滤，需要看到更多信息时调整它即可

### 命名

| 元素           | 规范                  |
|--------------|---------------------|
| 类 / 结构体      | `PascalCase`        |
| 方法 / 自由函数    | `snake_case`        |
| 私有成员         | `snake_case` 后缀 `_` |
| 公有成员 / 局部变量  | `snake_case`        |
| 命名空间         | `snake_case`        |
| 枚举值 (scoped) | `PascalCase`        |
| 常量           | `kPascalCase`       |
| 宏            | `SCREAMING_CASE`    |

### 命名空间

与架构层次对应：

| 命名空间                 | 层次         | 职责                           |
|----------------------|------------|------------------------------|
| `qualquer::optix`    | OptiX 封装层  | Context、Pipeline、AS、Denoiser |
| `qualquer::vulkan`   | Vulkan 呈现层 | Swapchain、interop            |
| `qualquer::renderer` | 渲染逻辑层      | PT 管线、材质、场景                  |
| `qualquer::app`      | 应用层        | 窗口、输入、加载、UI                  |

## 工作区改动处理

### IDE 删 include

工作区出现的 include 删除是 IDE 基于传递性包含分析的结果。**编译/链接失败前不主动恢复**——“显式依赖更卫生”是揣测，不构成恢复依据。

### 格式化与 tidy 是用户意图

工作区中的格式化与 tidy 应用均为**用户主动操作**的产物，代表用户意图，不视为自动污染。默认一并接受；质疑须有实际理由，不得凭样式偏好逐个横挑。

### 改动归属判断

工作区出现 agent 不记得自己执行的改动时，不得假设来源。归属不明时提交前须向用户确认，不得擅自并入 commit。

### 局部回退须限定范围

`git checkout -- <file>` 会无差别丢弃该文件**全部**未提交内容，用于“回退局部某行”时属越权。如需精准回退，用 patch
模式或手动编辑只移除真正要撤的几行。

## 设计原则

### 所有权

**原则**：句柄与资源的单一所有者，使用方不缓存副本。

每个资源只有一个所有者（owner）。其他对象作为使用方，在 init/destroy/recreate 等操作时从调用方接收所有者的引用，**不**
在自身成员中缓存任何由所有者拥有的句柄或引用。

**理由**：

- 单一来源：句柄只存在一处，避免多份拷贝因句柄重建/替换而过期
- 句柄缓存开销可忽略，按所有权清晰设计，非性能取舍

### 状态与输入分离

**原则**：成员表示对象的客观状态，不充当某次操作的隐式输入。

操作的输入走参数，输出可写回成员。区分标准是：该值属于「对象现在是什么」（状态）还是「这次调用想要什么」（请求）。前者读成员合法，后者必须走参数。

**理由**：

- 成员兼作输入通道时，调用方需先改成员再调用，职责混同、隐藏了调用对成员状态的依赖
- 输入走参数后，函数签名本身声明了依赖；状态成员也保持单一来源

## 项目结构

每层独立 CMake static library，编译期强制单向依赖（上层依赖下层，下层不知道上层的存在）。

```
        app
         ↓
      renderer
       ↓   ↓
   optix   vulkan
```

### 第三方库集成规范

vcpkg 有端口的库通过 `vcpkg.json` 管理。vcpkg 无端口的库放入 `third_party/<name>/`，每个库自带 `CMakeLists.txt`，顶层
`CMakeLists.txt` 通过 `add_subdirectory()` 引入，消费方只需 `target_link_libraries()` 链接目标名。

| 类型     | 目录结构                                 |
|--------|--------------------------------------|
| 源码编译   | `include/qualquer/<name>/` + `src/`  |
| 预编译二进制 | `include/` + `lib/` + `bin/`（保持上游布局） |

源码编译库的头文件放入 `include/qualquer/<name>/` 以统一 include 路径风格（`#include <qualquer/<name>/xxx.h>`）。PRIVATE
include 指向头文件目录，解决源文件中的相对路径引用。第三方代码统一禁用编译器警告（MSVC `/W0`）。
